# Certificate Management Module

## 1. Overview

**Description**  
The Certificate Management Module manages all **patient-related official certificates** generated by the hospital, including **Discharge Certificates**, **Birth Certificates**, **Death Certificates**, and **Transfer Certificates**. It standardizes certificate content and workflows, enforces authorization and legal requirements, and produces printable, traceable documents.

**Objectives**

- **Standardize documentation** for all patient-facing certificates.
- **Ensure legal and regulatory compliance** (especially for birth and death registration rules in the local jurisdiction).
- **Enforce doctor authorization** and role-based approval chains.
- **Provide auditability** for all certificate content and lifecycle events.
- **Support configurable print layouts** with hospital branding, certificate numbers, and optional QR / digital signatures.

---

## 2. Pre‑Requisites & Scope

### 2.1 Pre‑Requisite Modules

- **Patient Registration / Patient Master**
  - Source of truth for Patient ID, Name, Demographics, and core identifiers.
- **Admission – IPD Module**
  - Admission number, ward/bed/cabin details, admission/discharge dates, treating doctors.
- **Doctor Module**
  - Doctor master (names, codes, registration numbers, specialties).
- **Hospital Operations – System Settings**
  - Organization profile (hospital name, logo, address).
  - Certificate numbering and print template configuration (via Printing & Barcode settings).
  - Role and permission configuration.
- **Billing / IPD Billing (for Discharge & Transfer)**
  - Final discharge confirmation and clearance status (optional validation).

### 2.2 Module Scope

- A single consolidated **Certificate Management** module, with subtypes:
  - **Discharge Certificate**
  - **Birth Certificate**
  - **Death Certificate**
  - **Transfer Certificate**
  - **ICU Discharge Certificate**
  - **NICU Discharge Certificate**
  - **NICU Transfer Note**
- For each certificate type, provide:
  - Structured UI forms for data entry and review.
  - Status lifecycle (Draft → Pending Approval → Approved → Printed → Cancelled).
  - Print preview and printing with configurable templates.
  - Role-based access and approvals.
  - Complete audit trail of changes, approvals, and prints.

Out of scope (for this document):

- Jurisdiction-specific integration with civil registration systems (can be defined separately if needed).

## 3. Certificate Types & Core Data Elements

### 3.1 Discharge Certificate

#### 3.1.1 Purpose

Issued when a patient is **discharged after treatment** from IPD or day care. Summarizes the case history, treatment course, and follow-up plan.

#### 3.1.2 Patient & Administrative Information

- **Patient ID \***
  - Type: Search / text.
  - Source: Patient Master.
  - Validation: Must match an existing patient.
- **Patient Name (auto)**
  - Type: Read-only text.
  - Auto-populated from Patient Master when Patient ID is selected.
- **Admission No \***
  - Type: Dropdown / text (lookup).
  - Source: Active or recently closed admission for the patient.
  - Validation:
    - Must match an existing IPD admission.
- **Type of Discharge \***
  - Type: Dropdown.
  - Examples: Recovered, Improved, LAMA, Referred, Death.
  - Validation: Must be configured in Discharge Type master.
- **Discharge Date \***
  - Type: Date (and optional time).
  - Validation:
    - Must be ≥ Admission Date.
    - Must not be in the future beyond allowed tolerance (configurable).
- **Ward / Bed / Cabin**
  - Type: Read-only text or dropdown (from Admission/Bed Management).
  - Shows last occupied bed details at discharge.
- **Weight**
  - Type: Decimal.
  - Validation: ≥ 0 (optional).
- **Blood Group**
  - Type: Dropdown (from Blood Group master).
  - Optional if captured elsewhere and auto-filled.
- **Referred To**
  - Type: Text / dropdown (external facility names).
  - Optional; used when patient is referred out.

#### 3.1.3 Medical Details

- **Diagnosis \***
  - Type: Textarea or structured diagnosis picker (ICD-10/ICD-11).
  - Validation: Required; at least one final diagnosis.
- **Complaint on Admission \***
  - Type: Textarea.
  - Validation: Required.
- **Case Summary \***
  - Type: Textarea (multi-line).
  - Content: Brief history, hospital course, and important findings.
  - Validation: Required.
- **Investigations**
  - Type: Textarea or linked list of investigations.
  - Optional; may auto-pull from LIS/RIS orders and results where integrated.
- **Treatment Given \***
  - Type: Textarea.
  - Validation: Required; should summarize major treatments and procedures.
- **Follow Up**
  - Type: Textarea.
  - Optional but recommended; may contain review date, OPD clinic name, and special instructions.

#### 3.1.4 Treatment on Discharge (Medication Section)

- **Treatment on Discharge (section header)**
  - Non-editable label indicating start of discharge medication section.
- **Medicine Type \***
  - Type: Multi-select dropdown.
  - Values (configurable): Medicine, Fluid, Antibiotic, General, Others.
  - Validation: At least one type selected if any medication is prescribed.
- **Discharge Medication Line Items**
  - **Medicine Name \***
    - Type: Searchable dropdown / text.
    - Source: Product/Medicine master.
    - Validation: Must exist in product master unless free-text override is explicitly allowed for certain roles.
  - **Dose \***
    - Type: Text (e.g., 1+0+1, 5 ml BD).
    - Validation: Cannot be empty.
  - **Days \***
    - Type: Integer.
    - Validation: Must be ≥ 1.

#### 3.1.5 Consultant & Advice Section

- **Consultant Code**
  - Type: Auto-generated / read-only text.
  - Source: Doctor Master (primary consultant for the admission).
  - Validation: Must refer to an active doctor.
- **Consultant Name \***
  - Type: Text / dropdown (doctor lookup).
  - Validation: Required; must exist in Doctor Master.
- **Advice on Discharge**
  - Language options:
    - Bengali, English (configurable to support multiple templates).
  - Content:
    - Doctor can select from advice templates or enter free text.

### 3.2 Birth Certificate

#### 3.2.1 Purpose

Issued for **newborn babies** after delivery to document key birth details for clinical and legal records.

#### 3.2.2 Mother & Admission Information

- **Mother Patient ID \***
  - Type: Search / text.
  - Validation: Must exist in Patient Master.
- **Mother Name (auto)**
  - Type: Read-only text.
  - Auto-populated from Patient Master.
- **Admission No \***
  - Type: Dropdown / text.
  - Source: Mother’s active or recent IPD admission.
  - Validation: Must match a valid admission for the mother.
- **Ward / Bed / Cabin \***
  - Type: Dropdown / text.
  - Validation: Must exist in Bed Master and be linked to the admission.

#### 3.2.3 Baby Information

- **Name of the Baby \***
  - Type: Text.
  - Validation: Required (if unnamed, a placeholder per hospital policy may be allowed).
- **Gender \***
  - Type: Dropdown.
  - Values: Male, Female, Other.
- **Birth Date & Time \***
  - Type: DateTime.
  - Validation:
    - Must be ≤ system current date/time.
    - Must be within the range of the mother’s admission period (configurable tolerance).
- **Birth Weight \***
  - Type: Decimal (kg).
  - Validation: ≥ 0.
- **Delivery Type \***
  - Type: Dropdown.
  - Values: Normal, C-section, Assisted (forceps/vacuum – optional extension).
- **Multiple Birth \***
  - Type: Dropdown (Yes/No).
  - Default: No.
  - If Yes:
    - System should allow entry of sibling baby records (e.g., Baby 1, Baby 2, etc.) linked to the same delivery.

#### 3.2.4 Medical Information

- **APGAR Score**
  - Type: Integer.
  - Validation: 0–10 (may capture at 1 and 5 minutes if extended).
- **Complications**
  - Type: Textarea.
  - Optional; free-text description of delivery complications.
- **Neonatal Status \***
  - Type: Dropdown / text.
  - Values: Live, Stillborn, Critical (configurable).

#### 3.2.5 Doctor Authorization

- **Obstetrician Name \***
  - Type: Dropdown / text.
  - Validation: Must exist in Doctor Master and be eligible to sign birth certificates per policy.
- **Pediatrician Name \***
  - Type: Dropdown / text.
  - Validation: Must exist in Doctor Master.

### 3.3 Death Certificate

#### 3.3.1 Purpose

Issued after **patient death is medically confirmed** to document cause, time, and circumstances of death as required by law and hospital policy.

#### 3.3.2 Patient & Admission Information

- **Patient ID \***
  - Type: Search / text.
  - Validation: Must exist in Patient Master.
- **Name \***
  - Type: Text (auto-filled but editable if policy allows localization).
- **Age**
  - Type: Integer.
  - Validation: ≥ 0.
- **Gender \***
  - Type: Dropdown.
  - Values: Male, Female, Other.
- **Admission No \***
  - Type: Dropdown / text.
  - Validation: Must match active or last admission record for the patient.
- **Ward / Bed / Cabin \***
  - Type: Dropdown / text.
  - Validation: Must be valid bed/ward at time of death.

#### 3.3.3 Death Details

- **Date & Time of Death \***

  - Validation:
    - ≤ system current date/time.

- **Place of Death \***
  - Type: Dropdown.
  - Values: ICU, Ward, OT, Emergency, Other (configurable).
- **Cause of Death (Primary) \***
  - Type: Text / structured ICD-coded field.
  - Validation: Required.
- **Secondary Cause**
  - Type: Text.
  - Optional; contributing causes or comorbidities.
- **Duration of Illness**
  - Type: Text / numeric.
  - Optional; examples: “2 months”, “3 days”.

#### 3.3.4 Certification & Authorization

- **Attending Doctor \***
  - Type: Dropdown / text.
  - Validation: Must exist in Doctor Master.
- **Verified By \***
  - Type: Dropdown / text.
  - Validation: Must exist in Doctor Master; may be same as Attending or a designated verifier per policy.
- **Medical Registration Number \***
  - Type: Text.
  - Validation:
    - Must match registration number of selected verifying doctor.

## 4. Common Features & Business Rules (All Certificates)

### 4.1 Status Lifecycle

- **Draft**
  - Created but not yet sent for medical approval.
  - Editable by initiating role (e.g., Nurse, Registrar).
- **Pending Approval**
  - Submitted to responsible doctor for review.
  - Editable only by designated approver.
- **Approved**
  - Medically and administratively approved.
  - Content locked except for limited correction workflows (with audit).
- **Printed**
  - At least one official print has been generated.
  - System records print timestamp, user, and printer/location.
- **Cancelled**
  - Certificate logically voided (e.g., error, duplicate).
  - Requires reason and appropriate approval.

Allowed transitions (configurable):

- Draft → Pending Approval.
- Pending Approval → Approved.
- Approved → Printed.
- Approved → Cancelled.
- Printed → Cancelled (with stricter approval and traceability).

### 4.2 Role-Based Permissions (High-Level)

- **Nurse / Ward Desk**
  - Create and edit **Draft** certificates.
  - Submit to doctor for approval.
- **Doctor / Consultant**
  - Review, edit medical content.
  - Approve or reject certificates (Discharge, Death, Transfer; Birth as applicable).
- **Admin / Medical Records Officer**
  - Final approval in special cases.
  - Cancel certificates with mandatory reason and audit.
  - Manage numbering and template assignments (via System Settings).
- **Reception / Front Desk**
  - View and **Print** Approved certificates.
  - Cannot change medical content.

Exact permissions shall be configured in **User Management / Hospital Operations** (role–menu–action matrix).

### 4.3 Printing & Layout

- **Print Layout Configuration**
  - Hospital header with logo, name, address, contact.
  - Certificate title (e.g., “Discharge Certificate”, “Birth Certificate”).
  - **Certificate Number** (auto-generated from a configurable series).
  - Patient and certificate content blocks as defined above.
  - Footer with disclaimers and hospital notes.
- **QR Code (Optional)**
  - QR code including:
    - Certificate ID.
    - Hash or URL to verify authenticity (future enhancement).
- **Signatures**
  - Space for handwritten signatures where necessary.
  - Optional **digital signature** fields (if implemented) linked to doctor accounts.

### 4.4 Audit & Compliance

- Maintain audit trail for:
  - Create, update, approve, cancel, and print actions.
  - Before/after values for core medical and patient-identifying fields.
- Logs must record:
  - User, timestamp, action, and reason (where applicable).
- For Death and Birth Certificates:
  - System should support data extraction for statutory reporting (e.g., CSV/PDF export for civil registration authorities).

## 5. Workflows

### 5.1 Discharge Certificate Workflow

1. Discharge confirmed in IPD Billing/Admission.
2. Nurse/ward desk creates **Draft Discharge Certificate** from admission record.
3. Doctor reviews, edits medical content, and sets status to **Approved**.
4. Reception or records prints certificate; status records **Printed**.
5. Any corrections after print require:
   - New version or cancellation of previous certificate with reason.

### 5.2 Birth Certificate Workflow

1. Delivery completed and recorded in IPD/NICU workflow.
2. Authorized user opens **Birth Certificate** for the mother’s admission.
3. Enter/verify baby details and doctor authorizations.
4. Doctor approves.
5. Certificate is printed and, where required, information is queued for civil registration reporting.

### 5.3 Death Certificate Workflow

1. Death is recorded in clinical documentation and marked in Admission/IPD.
2. Treating/attending doctor enters death details and cause(s).
3. Verifying doctor confirms and sets certificate to **Approved**.
4. Admin/records validates and prints, handing certificate to relatives as per policy.

### 5.4 Transfer Certificate Workflow

1. Decision to transfer patient is taken and documented.
2. Physician/Nurse creates **Transfer Certificate** from active admission.
3. Attending and verifying doctors review clinical status and referral reason.
4. Certificate is approved and printed to accompany the patient to the receiving facility.

### 5.5 ICU Discharge Certificate Workflow

1. Patient is discharged from ICU (to ward, home, or referral).
2. Nurse/ward desk or ICU staff creates **ICU Discharge Certificate** from admission record.
3. Intensivist or attending consultant(s) review condition on admission vs. discharge, investigations, and treatment.
4. Doctor approved; certificate is printed.
5. Same status lifecycle (Draft → Pending Approval → Approved → Printed) as other certificates.

### 5.6 NICU Discharge Certificate Workflow

1. Neonate is discharged from NICU (to mother's ward, home, or referral center).
2. NICU staff creates **NICU Discharge Certificate** from admission record.
3. Neonatologist/Pediatrician reviews birth history, vital parameters, special medications, and follow-up plan.
4. Doctor approved; certificate is printed.

### 5.7 NICU Transfer Note Workflow

1. Decision to transfer neonate from NICU to ward or another facility is taken.
2. NICU staff creates **NICU Transfer Note** from active admission.
3. Attending consultant(s) review clinical status and investigations at transfer.
4. Note is approved and printed to accompany the baby during transfer.

## 6. Functional Requirements (FR‑Cx.y)

> **Note**: These functional requirements reference the detailed field definitions and workflows in Sections 3–5. Prefix `FR‑C` indicates **Certificate Management**.

### 6.1 Discharge Certificate – Functional Requirements

- **FR‑C1.1 – Discharge Certificate Creation & Patient Context**
  - System shall allow authorized users to **create a Discharge Certificate** for a selected IPD/day‑care admission, using:
    - Patient and admission search as defined in 3.1.2.
    - Auto‑population of patient demographics, admission dates, and bed details.
  - System shall **prevent certificate creation** if:
    - No matching admission is found, or
    - Admission is not in a discharge‑eligible status (configuration driven).

- **FR‑C1.2 – Capture of Administrative & Clinical Fields**
  - System shall capture all fields listed in 3.1.2 and 3.1.3 with:
    - Required/optional flags enforced in UI.
    - Field‑level validation rules (e.g., Discharge Date ≥ Admission Date, mandatory Diagnosis, Case Summary, Treatment Given).
  - System shall support:
    - Free‑text entry for narrative fields.
    - Optional linkage to coded diagnoses (ICD) and investigations.

- **FR‑C1.3 – Treatment on Discharge (Medication Lines)**
  - System shall support adding one or more **discharge medication lines** as in 3.1.4:
    - Medicine Type, Medicine Name, Dose, Days.
  - System shall validate:
    - At least one Medicine Type when lines exist.
    - Medicine Name against Product/Master (unless override allowed).
    - Days ≥ 1.

- **FR‑C1.4 – Consultant & Advice Section**
  - System shall display and store Consultant Code/Name and Advice on Discharge as defined in 3.1.5.
  - System shall support:
    - Selection of standard advice templates per language.
    - Free‑text editing by authorized doctors.

### 6.2 Birth Certificate – Functional Requirements

- **FR‑C2.1 – Birth Certificate Creation Linked to Mother Admission**
  - System shall allow creation of a **Birth Certificate** only when a valid mother admission is selected as per 3.2.2.
  - System shall auto‑populate mother details (ID, Name) and validate that:
    - Admission belongs to the selected mother.
    - Ward/Bed/Cabin exists and is linked to that admission.

- **FR‑C2.2 – Capture of Baby & Medical Information**
  - System shall capture all baby and delivery‑related fields in 3.2.3 and 3.2.4, enforcing:
    - Mandatory fields (Name of Baby, Gender, Birth Date & Time, Delivery Type, Neonatal Status).
    - Validation of APGAR Score range and date/time constraints.
  - For **Multiple Birth = Yes**, system shall allow:
    - Creation of multiple baby records tied to the same delivery/admission.

- **FR‑C2.3 – Doctor Authorization**
  - System shall require Obstetrician and Pediatrician entries as in 3.2.5.
  - Selected doctors must:
    - Exist in Doctor Master.
    - Meet any additional policy‑based eligibility criteria for signing Birth Certificates.

### 6.3 Death Certificate – Functional Requirements

- **FR‑C3.1 – Death Certificate Creation & Linkage**
  - System shall allow **Death Certificate** creation only when:
    - Patient exists in Patient Master.
    - A relevant admission exists (as per 3.3.2).
  - System shall enforce:
    - Single official death certificate per admission by default, unless overrides are allowed for corrections.

- **FR‑C3.2 – Capture of Death Details**
  - System shall capture all fields in 3.3.3 with:
    - Validation of Date & Time of Death (≤ current time, ≥ admission time).
    - Mandatory primary cause of death.
  - System shall support separate storage of:
    - Primary vs secondary causes.
    - Duration of illness as structured or free text.

- **FR‑C3.3 – Certification & Medical Authorization**
  - System shall require Attending Doctor, Verified By, and Medical Registration Number as per 3.3.4.
  - System shall:
    - Validate that Medical Registration Number matches the verifying doctor.
    - Record both attending and verifying doctors in the audit trail.

### 6.4 Transfer Certificate – Functional Requirements

- **FR‑C4.1 – Transfer Certificate Creation from Active Admission**
  - System shall allow **Transfer Certificate** creation only from an **active admission** (3.4.2).
  - Patient, Admission, and Ward/Bed/Cabin information must be auto‑populated and read‑only.

- **FR‑C4.2 – Capture of Transfer Details**
  - System shall capture fields in 3.4.3:
    - Date & Time of Transfer.
    - Referral Reason.
    - Receiving Facility.
  - System shall enforce:
    - Transfer date/time validation similar to 3.3.3.
    - Mandatory Referral Reason.

- **FR‑C4.3 – Transfer Certification**
  - System shall require Attending Doctor, Verified By, and Medical Registration Number, with validations as in 3.4.4.
  - System shall store transfer‑specific certification separately from Death Certification.

### 6.5 ICU Discharge Certificate – Functional Requirements

- **FR‑C5.1 – ICU Discharge Certificate Creation**
  - System shall allow creation of **ICU Discharge Certificate** for patients discharged from ICU, using Patient ID and Admission No as per 3.5.2.
  - System shall capture condition on admission and condition on discharge (3.5.4), key investigations (3.5.5), and treatment on discharge with medication lines.
  - System shall support multiple consultant entries (3.5.5).

### 6.6 NICU Discharge Certificate – Functional Requirements

- **FR‑C6.1 – NICU Discharge Certificate Creation**
  - System shall allow creation of **NICU Discharge Certificate** for neonates discharged from NICU, using Patient ID (baby) and Admission No as per 3.6.2.
  - System shall capture Guardian Name, Birth History, Vital Parameters at Discharge, Special Medication, and treatment on discharge (3.6.3).
  - System shall support multiple consultant entries.

### 6.7 NICU Transfer Note – Functional Requirements

- **FR‑C7.1 – NICU Transfer Note Creation**
  - System shall allow creation of **NICU Transfer Note** for neonates transferred from NICU, using Patient ID and Admission No as per 3.7.2.
  - System shall capture Transfer Date/Time, clinical summary, investigations at transfer, and Note (handover instructions).
  - System shall support multiple consultant entries.

### 6.8 Common Certificate Features – Functional Requirements

- **FR‑C8.1 – Status Lifecycle Management**
  - System shall implement the status lifecycle described in 4.1 (Draft, Pending Approval, Approved, Printed, Cancelled) and enforce allowed transitions.
  - Status changes shall:
    - Be restricted to appropriate roles.
    - Be captured in the audit log with user, time, and optional reason.

- **FR‑C8.2 – Role‑Based Access Control**
  - System shall map permissions in 4.2 to roles configured in User Management:
    - Draft creation/editing for nursing/ward users.
    - Medical approval for doctors.
    - Administrative overrides/cancellation for Admin/MR.
    - Print‑only access for Reception/Front Desk.

- **FR‑C8.3 – Printing & Numbering**
  - System shall generate **Certificate Numbers** from configurable series per certificate type, as referenced in 4.3.
  - System shall log every print event (who, when, which printer/template).

- **FR‑C8.4 – Audit & Reporting**
  - System shall maintain an audit log as in 4.4 for all certificate types.
  - System shall support extraction of Birth and Death certificate data for statutory reporting in machine‑readable formats (e.g., CSV).

## 7. Architectural Considerations

- **Single Module, Multiple Types**
  - Implement Certificate Management as one module with:
    - A shared certificate header (ID, patient, admission, status, type).
    - Type-specific payload sections (Discharge, Birth, Death, Transfer).
- **Shared Services**
  - Common services for:
    - Number generation.
    - Status transitions and approvals.
    - Printing and export.
    - Audit logging.
- **Integration Points**
  - Read-only integration with:
    - Patient Master (for IDs and demographics).
    - Admission/IPD (admission status, ward/bed, discharge/death flags).
    - Doctor Master (for authorization and signatures).
  - Optional integration with:
    - Reporting/Analytics for regulatory reports.
    - External civil registry APIs (future enhancement).

### 8.2 Birth Certificate – Create / Edit Screen

- **Header Region**
  - Mother context:
    - Mother Patient ID, Name, Age, Gravida/Para (optional, if available).
    - Admission No, Ward/Bed/Cabin, Admission/Delivery Dates.
- **Main Form – Baby Details**
  - Baby information block from 3.2.3:
    - Name of Baby, Gender, Birth Date & Time, Birth Weight, Delivery Type, Multiple Birth.
  - For multiple births:
    - Tabbed or list UI for Baby 1, Baby 2, etc.
- **Medical & Authorization Section**
  - APGAR Score, Complications, Neonatal Status (3.2.4).
  - Obstetrician and Pediatrician fields (3.2.5).
- **Actions**
  - `Save Draft`, `Send to Doctor`, `Approve`, `Print`.

### 8.3 Death Certificate – Create / Edit Screen

- **Header Region**
  - Patient context similar to Discharge:
    - Patient ID, Name, Age, Gender, Admission No, Ward/Bed/Cabin.
- **Main Form**
  - Left side: Patient/Admission and Death Details (3.3.2, 3.3.3).
  - Right side: Certification/Authorization block (3.3.4).
- **Visual Emphasis**
  - Clear labeling of Death Date & Time and primary cause of death.
  - Warning banner indicating legal sensitivity and restricted access.

### 8.4 Transfer Certificate – Create / Edit Screen

- **Header Region**
  - Active admission banner identical to Discharge, but with prominent “Transfer” title.
- **Main Form**
  - Transfer Details (3.4.3) – date/time, referral reason, receiving facility.
  - Certification block with attending and verifying doctors (3.4.4).
- **Actions**
  - `Save Draft`, `Send for Approval`, `Approve`, `Print`.

### 8.5 ICU Discharge Certificate – Create / Edit Screen

- **Header Region** – Patient context (Patient ID, Name, Age, Bed No, Admission/Discharge Dates).
- **Main Form** – Clinical Summary (3.5.3), Condition on Admission & Discharge (3.5.4), Key Investigations (3.5.5), Treatment on Discharge, Hold Drugs, Note.
- **Consultant Section** – Multiple consultants (Dr Code, Dr Name).
- **Actions** – `Save Draft`, `Send for Approval`, `Approve`, `Print`.

### 8.6 NICU Discharge Certificate – Create / Edit Screen

- **Header Region** – Baby/Neonate context (Patient ID, Name, Guardian, Weight, Admission/Discharge Dates).
- **Main Form** – Clinical Summary (3.6.3), Vital Parameters, Special Medication, Treatment on Discharge.
- **Consultant Section** – Multiple consultants.
- **Actions** – `Save Draft`, `Send for Approval`, `Approve`, `Print`.

### 8.7 NICU Transfer Note – Create / Edit Screen

- **Header Region** – Baby/Neonate context with Transfer Date/Time.
- **Main Form** – Presenting Complaint, Final Diagnosis, Note (handover instructions), Investigations at Transfer.
- **Consultant Section** – Multiple consultants.
- **Actions** – `Save Draft`, `Send for Approval`, `Approve`, `Print`.

### 8.8 Certificate List / Search Screen

- **Filters**
  - Date range, Certificate Type, Status, Patient ID/Name, Doctor, Ward/Department.
- **Grid Columns**
  - Certificate No, Type, Patient, Admission No, Status, Created On, Approved On, Printed On, Last Updated By.
- **Row Actions**
  - `View`, `Edit` (subject to status/role), `Print`, `Cancel` (with appropriate permission and reason).
